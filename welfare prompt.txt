

─── PHASE 1 OF 8 ───
Project Foundation, Design System & Authentication

What we are building
A web application called the CXT WELFARE— an internal financial management tool for a team's welfare account. It handles member contributions, welfare events, expense tracking, receipt storage, real-time reporting, and role-based access control. It is not a public product. Every user account is created by an administrator — there is no self-registration.

Design Language
Study the following design direction carefully and apply it consistently across every screen you build:
Layout pattern:
	•	Split-panel login page: form on the left (~45% width), a full-height image or branded panel on the right (~55%). Clean, uncluttered.
	•	After login: fixed left sidebar navigation with icon + label, top header bar with search, notifications bell, and user profile. Main content area to the right.
	•	Dashboard uses a summary card row at the top (3–4 stat cards side by side), followed by a wider content area with charts, tables, and action panels below.
Colour system — blue only, all shades must complement each other:
Primary Dark Blue:     #0F2D5C   (sidebar, headers, primary buttons)
Primary Mid Blue:      #1A56DB   (active states, links, CTA buttons)
Accent Blue:           #3B82F6   (highlights, icons, progress indicators)
Light Blue:            #DBEAFE   (card backgrounds, hover states, table row alt)
Pale Blue:             #EFF6FF   (page backgrounds, input backgrounds)
White:                 #FFFFFF   (card surfaces, modal backgrounds)
Text Dark:             #0F172A   (headings, primary body text)
Text Mid:              #475569   (secondary text, labels, metadata)
Text Light:            #94A3B8   (placeholders, disabled states, captions)
Border:                #CBD5E1   (input borders, dividers, table borders)
Success:               #16A34A   (paid status, success states — use sparingly)
Warning:               #D97706   (pending, missing receipts — use sparingly)
Danger:                #DC2626   (overdue, lockout, error states — use sparingly)
Typography:
	•	Font: Inter (Google Fonts). Import it.
	•	Headings: font-weight 700. Page titles 24–28px. Section headings 18–20px.
	•	Body: font-weight 400, 14–15px.
	•	Labels and metadata: font-weight 500, 12–13px, uppercase tracking on table headers.
Component style:
	•	Cards: white background, 1px solid #CBD5E1 border, border-radius 12px, subtle box-shadow (0 1px 4px rgba(0,0,0,0.06)).
	•	Buttons: Primary = #1A56DB background, white text, border-radius 8px, font-weight 600. Secondary = white background, #1A56DB border and text. Danger = #DC2626.
	•	Inputs: border 1px #CBD5E1, border-radius 8px, background #EFF6FF, focus ring #1A56DB, 40px height.
	•	Sidebar: #0F2D5C background, white text, active item has #1A56DB left-border accent and slightly lighter background tint. Icons are clean and minimal.
	•	Stat cards on dashboard: white background, coloured icon container top-left (use light blue tint), metric value large and bold in #0F2D5C, label below in #475569.
	•	Tables: clean, no heavy borders. Header row background #DBEAFE, text #0F2D5C font-weight 600 uppercase 12px. Alternating rows: white / #EFF6FF. Row hover: #DBEAFE.
	•	Status badges: pill-shaped, small. Paid = light green bg / green text. Pending = light amber bg / amber text. Overdue = light red bg / red text. Read-only = light blue bg / blue text.
	•	Modals: white card, border-radius 16px, backdrop blur, header with title and close X, footer with action buttons right-aligned.
	•	No ai generated like ui like  emojis…etc

─── PHASE 7 OF 8 ───
Supabase Backend Integration

Phase 7 Task — Connect Everything to Supabase

Supabase Project Setup
Connect the application to a Supabase project. Use the Supabase JavaScript client (@supabase/supabase-js). Store the Supabase URL and anon key in environment variables:
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=

Database Schema
Create the following tables in Supabase. Apply Row Level Security (RLS) to every table.
-- Users / Profiles (extends Supabase auth.users)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name TEXT NOT NULL,
  member_id TEXT UNIQUE,
  email TEXT NOT NULL,
  phone TEXT,
  role TEXT NOT NULL CHECK (role IN ('super_admin','treasurer','chairman','secretary','auditor','member')),
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','disabled')),
  must_change_password BOOLEAN DEFAULT TRUE,
  failed_login_attempts INT DEFAULT 0,
  locked_until TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  created_by UUID REFERENCES profiles(id)
);

-- Members (welfare members who contribute — same as profiles but this captures contribution expectations)
CREATE TABLE contribution_settings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  monthly_amount NUMERIC(12,2) NOT NULL,
  effective_from DATE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Monthly Contributions
CREATE TABLE contributions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id UUID NOT NULL REFERENCES profiles(id),
  month INT NOT NULL CHECK (month BETWEEN 1 AND 12),
  year INT NOT NULL,
  expected_amount NUMERIC(12,2) NOT NULL,
  amount_paid NUMERIC(12,2) DEFAULT 0,
  payment_date DATE,
  payment_method TEXT CHECK (payment_method IN ('cash','mobile_money','bank_transfer')),
  status TEXT DEFAULT 'unpaid' CHECK (status IN ('paid','partial','unpaid')),
  recorded_by UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(member_id, month, year)
);

-- Welfare Events
CREATE TABLE events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  event_type TEXT NOT NULL CHECK (event_type IN ('birthday','bereavement','meetup','end_of_year','hospital_support','team_bonding','other')),
  event_date DATE NOT NULL,
  description TEXT,
  organiser TEXT,
  expected_budget NUMERIC(12,2),
  status TEXT DEFAULT 'upcoming' CHECK (status IN ('upcoming','in_progress','completed','cancelled')),
  created_by UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Expenses
CREATE TABLE expenses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id),
  category TEXT NOT NULL CHECK (category IN ('food_drinks','cake','decoration','transport','venue','gifts','logistics','emergency_support','other')),
  description TEXT NOT NULL,

	•	Chairman: oversight access. Can approve/flag, manage users, view all reports. Cannot directly record financial entries.
	•	Super Admin: full access to everything.

2. Edge Case Handling
Implement proper handling for each of these scenarios:
	•	Partial payment: member pays GHS 30 when expected GHS 50. Status = Partial. Outstanding = GHS 20. Show this clearly on the row and in the member's history. The income posted is GHS 30, not GHS 50.
	•	Member pays two months at once: treasurer can open each month separately and mark payment for each. No bulk-month payment in one entry — each month is its own record.
	•	Contribution amount changes: new contribution_settings entry with a new effective_from date. Historical records keep their original expected_amount. Only new months after the effective date use the new amount.
	•	Disabled user's data: when an account is disabled, their records remain fully visible and attributed to them in all reports and the audit log. The name appears as "[Name] (Disabled)" in the audit log to indicate the account is no longer active.
	•	Expense without event: not possible — event field is required and the expense form will not submit without one.
	•	Receipt over 5MB: file upload validates size client-side and shows inline error "File too large. Maximum size is 5MB."
	•	Empty month: if a month has no contributions recorded yet, the report shows all members as Unpaid with GHS 0 collected — not an empty/broken state.
	•	Zero balance edge case: if expenses exceed contributions (balance goes negative), the balance card on the dashboard shows the figure in red with a warning label "Balance is negative."
